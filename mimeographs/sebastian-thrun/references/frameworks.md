# Frameworks

These frameworks represent Thrun's operational methods for engineering, innovation, and research.

## Continuous Weak-Link Testing (Early Freeze)
A systems engineering framework for high-stakes deadlines: freeze development early and focus entirely on breaking and fixing the weakest link.
1. Freeze all software development a full month before the deadline.
2. Build a dedicated testing team with a comprehensive testing booklet.
3. Run daily tests to identify vulnerabilities.
4. Continuously focus engineering efforts exclusively on improving the weakest part of the system.
> "It was a continuous focus on improving the weakest part of the system. And as long as you you focus on improving the weakest part of the system, you eventually build a really great system."
*(sources: src_029, src_016)*

## End-to-End System Building
A method for prioritizing engineering efforts by building a complete system immediately and letting real-world failures dictate the roadmap.
1. Build an end-to-end system literally from day one.
2. Test it immediately in the real world (e.g., driving on public streets).
3. Let the system fail to identify the actual points of failure.
4. Isolate the top most important problems revealed by the failure.
5. Solve those specific problems instead of arguing over hypothetical component features.
*(sources: src_015)*

## Moonshot Ideation
A framework for inventing massively impactful, audacious technologies by starting with a deep personal or societal problem.
1. Pick a problem you deeply care about (e.g., a personal loss or societal issue).
2. Ask yourself if you can envision a technology that can solve this problem.
3. Work on it, trusting that technologically it can likely be solved if you try hard enough.
> "So the first thing is pick a problem you care about like your grandmother cares about like in my case the death of my friend Harold and then ask yourself can you envision a technology that can solve this problem and then work on it."
*(sources: src_020)*

## Probabilistic State Estimation (Bayes Filter)
A framework for inferring unobservable quantities from noisy, partial sensor data in unpredictable environments.
1. Represent the robot's belief as a probability density function.
2. Update the belief using Bayes rule when a sensor measurement is taken.
3. Shift and smooth the belief when the robot moves to account for actuation uncertainty.
4. Repeat continuously.
*(sources: src_037, src_047)*

## Problem-First AI Research
A method for deciding what to research by starting with a real-world problem rather than trying to force mechanisms together.
1. Identify a specific problem in the world that you believe is solvable.
2. Ensure the problem inherently requires the specific mechanisms you want to test.
3. Attempt to solve the problem.
4. If solved and the mechanisms were indeed needed, you have a phenomenal research result.
*(sources: src_015)*

## The Grandmother Test
A test for ensuring a startup's value proposition is rooted in clear market demand rather than technical sophistication. Attempt to explain the startup's value proposition to a 'normal human being' (like a grandmother). Ensure the explanation focuses on useful value rather than technical sophistication.
*(sources: src_032)*
